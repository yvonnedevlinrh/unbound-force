### Requirement: Structured PR Body Sections

The `/finale` command Step 5c MUST generate a PR body
with the following sections in order:

1. `## Summary` тАФ what was done (from commit analysis)
2. `## How to Test` тАФ verification steps (from spec
   acceptance criteria or task descriptions if available,
   otherwise synthesized from the diff)
3. `## How to Demo` тАФ demo walkthrough for reviewers
4. `## Key Files Changed` тАФ file listing with brief
   descriptions of what changed in each

Each section MUST contain substantive content. Sections
MUST NOT be left empty or contain only placeholder text.
For trivial changes (e.g., single-line typo fixes),
sections with insufficient source material (e.g.,
How to Demo for a typo fix) SHOULD contain a brief
note explaining the change is trivial, rather than
fabricating content. The section MAY be reduced to a
single sentence in such cases.

#### Scenario: PR body with structured sections

- **GIVEN** a branch with commits ready for PR creation
- **WHEN** `/finale` generates the PR body
- **THEN** the body MUST contain `## Summary`,
  `## How to Test`, `## How to Demo`, and
  `## Key Files Changed` sections with content

#### Scenario: Minimal commit history

- **GIVEN** a branch with a single commit `fix: typo`

- **WHEN** `/finale` generates the PR body
- **THEN** sections with insufficient source material

### Requirement: Commit Message Attribution

The `/finale` command Step 3 MUST append AI attribution
to generated commit messages in two forms:

1. A git trailer: `Assisted-by: <model>`
2. A human-readable footer: `Generated with AI
   assistance (<model>)`

Where `<model>` is the model family name the agent is
running as (see Requirement: Model Identification
below).

Both MUST appear after the commit body, separated from
the body by a blank line. The trailer MUST follow git's
standard `Key: Value` format.

The attribution MUST be included in the proposed commit
message shown to the user for approval. The user MAY
edit or remove it during the approval step.

### Requirement: Model Identification

All commands that generate commit messages with AI
attribution (`/finale`, `/address-feedback`,
`/review-pr`) MUST resolve the `<model>` value to the
model family name the agent is currently running as.

The agent MUST extract the model family name by:
1. Reading the model identifier from the system prompt
   (e.g., "You are powered by the model named X. The
   exact model ID is Y.") or the runtime environment.
   If multiple sources provide a model identifier, the
   system prompt takes precedence.
2. Removing everything before and including the last
   `/` character (strips provider prefixes).
3. Removing everything after and including the first
   `@` character (strips routing suffixes).
4. Removing any trailing date suffix matching
   `-YYYYMMDD` тАФ a hyphen followed by exactly 8 digits
   (strips version date identifiers).
5. Repeatedly removing any trailing version segment
   matching `-N` тАФ a hyphen followed by a single digit
   at the end of the string тАФ until no more remain
   (strips generation and sub-version numbers, e.g.,
   `claude-opus-4-6` тЖТ `claude-opus`).
6. If the result is empty or whitespace-only, using the
   literal string `unknown-model`.

The resolved model name MUST be validated against the
pattern `[a-zA-Z0-9._-]+` before insertion into the
trailer. If the resolved name contains characters
outside this set (including whitespace, newlines, or
shell metacharacters), the agent MUST fall back to
`unknown-model`.

The model name MUST NOT include:
- Provider prefixes (e.g., `google-vertex-anthropic/`)
- Routing suffixes (e.g., `@default`)
- Version date suffixes (e.g., `-20250514`)
- Generation number suffixes (e.g., `-4`)
- Tool or product names (e.g., `OpenCode`)

Examples:
- `google-vertex-anthropic/claude-sonnet-4-20250514@default` тЖТ `claude-sonnet`
- `claude-opus-4-20250514` тЖТ `claude-opus`
- `gpt-4o` тЖТ `gpt-4o`
- `gemini-2.5-pro` тЖТ `gemini-2.5-pro`

If the agent cannot determine its model name, it MUST
use the literal string `unknown-model`. When the
`unknown-model` fallback is used, the command SHOULD
warn the user during the approval step (e.g., "Could
not determine AI model name тАФ using 'unknown-model'
in attribution").

#### Scenario: Model family name resolution

- **GIVEN** an agent powered by model
  `google-vertex-anthropic/claude-opus-4-20250514@default`
- **WHEN** generating the `Assisted-by` trailer
- **THEN** the trailer value MUST be `claude-opus`
  (model family only тАФ no provider prefix, routing
  suffix, date version, or generation number)

#### Scenario: Model name without date or generation

- **GIVEN** an agent powered by model `gpt-4o`
- **WHEN** generating the `Assisted-by` trailer
- **THEN** the trailer value MUST be `gpt-4o`
  (no stripping needed тАФ `4o` does not match
  a single-digit trailing pattern)

#### Scenario: Unknown model fallback

- **GIVEN** an agent that cannot determine its model
  name from system configuration
- **WHEN** generating the `Assisted-by` trailer
- **THEN** the trailer value MUST be `unknown-model`
- **AND** the command SHOULD warn the user during the
  approval step

#### Scenario: Model name with invalid characters

- **GIVEN** an agent whose system configuration reports
  a model name containing newline characters, colons,
  or shell metacharacters
- **WHEN** generating the `Assisted-by` trailer
- **THEN** the trailer value MUST be `unknown-model`

#### Scenario: Model name resolves to empty after
stripping

- **GIVEN** an agent whose full model identifier is
  `google-vertex-anthropic/` (no base name after the
  provider prefix)
- **WHEN** generating the `Assisted-by` trailer
- **THEN** the trailer value MUST be `unknown-model`

### Requirement: Review-Council Findings Section

When `/finale` generates a PR body, it SHOULD check the
conversation context for prior `/review-council` output.
If unresolved findings exist (findings that were
acknowledged but not fixed), `/finale` MUST include a
`## Known Issues` section in the PR body.

Each finding MUST include its severity and a brief
description. The section MUST appear between
`## Key Files Changed` and the attribution footer.

If no `/review-council` was run in the session, or no
unresolved findings exist, the section MUST be omitted.

#### Scenario: PR body with review-council findings

  description

### Requirement: PR Template Detection

The `/finale` command SHOULD detect a PR template by
checking for `.github/PULL_REQUEST_TEMPLATE.md` and
`.github/pull_request_template.md` (the two conventional
casing variants). Other casing variants are not
supported.

If a template is found, `/finale` MUST use the
template's section structure as the skeleton for the PR
body. Generated content MUST be mapped to template
sections using case-insensitive substring matching on
`##` headings:

| Generated Section | Matches Template Headings |
|---|---|
| Summary | Description, Summary, Overview, What |
| How to Test | Testing, Test, Test Plan, Verification |
| How to Demo | Demo, How to Demo |
| Key Files Changed | Files Changed, Changes |

Template sections that do not match generated content
MUST be preserved as-is for the user to fill in during
the approval step. Generated content for sections
without a template match MUST be appended at the end
of the template.

If no template is found, `/finale` MUST fall back to
the default structured format.

#### Scenario: Repository has PR template

- **GIVEN** a repository with
  `.github/PULL_REQUEST_TEMPLATE.md` containing
  `## Description` and `## Testing` sections
- **WHEN** `/finale` generates the PR body
- **THEN** the body MUST use the template's section
  structure, mapping generated summary content to
  `## Description` and test instructions to
  `## Testing`

#### Scenario: No PR template exists

- **GIVEN** a repository without a PR template file
- **WHEN** `/finale` generates the PR body
- **THEN** the body MUST use the default structured
  format (Summary, How to Test, How to Demo, Key Files
  Changed)

#### Scenario: Template exists but is malformed

- **GIVEN** a repository with
  `.github/PULL_REQUEST_TEMPLATE.md` that contains
  no `##` headings (empty, binary, or HTML-only)
- **WHEN** `/finale` attempts to parse the template
- **THEN** `/finale` MUST fall back to the default
  structured format and SHOULD warn the user that the
  template could not be parsed

## MODIFIED Requirements

### Requirement: Safe PR Body Passing

The `/finale` command Step 5e MUST use `--body-file`
instead of inline `--body` for `gh pr create`.

Previously: `gh pr create --title "<title>"
--body "<body>"`

The PR body MUST be written to a temporary file
(`mktemp`) with restrictive permissions (`chmod 600`).
The temp file MUST be passed via `--body-file <file>`.
The temp file MUST be cleaned up (`rm -f`) in all exit
paths (success, failure, user abort).

The `--title` flag MAY continue to use inline
interpolation since PR titles are single-line and
validated through user confirmation.

#### Scenario: PR creation with body-file

- **GIVEN** a PR body has been approved by the user
- **WHEN** `/finale` creates the PR via `gh pr create`
- **THEN** the body MUST be written to a temp file and
  passed via `--body-file`, not inline `--body`

#### Scenario: Temp file cleanup on failure

- **GIVEN** a PR body temp file has been created
- **WHEN** `gh pr create` fails
- **THEN** the temp file MUST be removed before
  reporting the error

#### Scenario: Temp file cleanup on user abort

- **GIVEN** a PR body temp file has been created
- **WHEN** the user declines the proposed PR content
  at the approval step
- **THEN** the temp file MUST be removed before
  stopping

#### Scenario: Temp file creation failure

- **GIVEN** temp file creation fails (mktemp fails)
- **WHEN** `/finale` attempts to write the PR body
- **THEN** `/finale` MUST report the error and STOP
  without falling back to inline `--body` interpolation
